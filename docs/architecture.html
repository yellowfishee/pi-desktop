pi-desktop 架构文档

基于 Tauri 的非侵入式 pi GUI 客户端 — 通过 stdin/stdout JSONL 协议与 pi CLI 进程通信

1. 整体架构

pi-desktop 的核心设计理念是非侵入式：不修改 pi 的任何代码，而是通过启动 pi --mode rpc 子进程，利用 stdin/stdout 的 JSONL 协议实现全部交互。

三层架构清晰分层：React 前端 负责 UI 渲染，Rust 后端 (Tauri) 负责进程管理和协议桥接，pi CLI 负责核心 AI 逻辑。

颜色含义说明
● 蓝色请求通道前端 invoke → Rust → pi stdin。也用于 extension_ui_response 回写
● 橙色响应通道pi stdout → Rust stdout_reader_task 读取
● 绿色事件通道Rust emit("pi-event") → 前端 listen → Zustand Store 更新
● 褐色内部通道oneshot channel 匹配 response id → 唤醒 send_command
● 紫色持久化~/.pi/agent/sessions/ (pi 自动写入) + config.json (Rust 写入)
● 红色异常进程退出/EOF → 清理状态 + emit("pi-process-exit") → 前端自动重启

2. 一次完整对话的数据流

1

TypeScript 用户点击发送 → sendCommand()

用户在 ChatPanel 输入消息，点击发送按钮。前端调用 sendCommand({ type:"prompt", message:"用户输入" })

底层通过 @tauri-apps/api invoke('send_command', { command }) 发起 Tauri IPC 调用

2

Rust send_command → stdin 写入 + 注册 oneshot

Rust commands::send_command() 接收到命令后：

1. 分配 id (如 "req-3a7f1b2c"), 若命令无 id 则自动生成
2. 创建 oneshot channel (tx, rx), 将 tx 存入 pending_commands HashMap
3. JSON 序列化命令 → 写入 pi 进程的 stdin: write_all(json + "\n")
4. flush stdin, 确保数据立即发送
5. await oneshot rx (带超时: prompt 180s, get_state 10s, 其他 30s)
3

pi stdout pi 处理命令 → 输出事件流

pi 核心引擎接收命令后，调用 LLM API，流式输出如下事件（每行一个 JSON）：

{ "type":"agent_start" }
{ "type":"message_update", "assistantMessageEvent":{"type":"text_delta","delta":"你好"} }
{ "type":"message_update", "assistantMessageEvent":{"type":"text_delta","delta":"，世界"} }
{ "type":"message_update", "assistantMessageEvent":{"type":"toolcall_start","partial":{"name":"read"}} }
{ "type":"message_update", "assistantMessageEvent":{"type":"toolcall_delta","delta":"..."} }
{ "type":"tool_execution_start", "toolCallId":"tc-1", "toolName":"read" }
{ "type":"tool_execution_end",   "toolCallId":"tc-1", "toolName":"read", "result":{...} }
{ "type":"message_update", "assistantMessageEvent":{"type":"text_delta","delta":"根据文件内容"} }
{ "type":"message_update", "assistantMessageEvent":{"type":"done","reason":"stop"} }
{ "type":"agent_end" }
{ "type":"response", "id":"req-3a7f1b2c", "success":true, "data":{...} }
4

Rust stdout_reader_task → 多路分派

Rust 后台 tokio 任务逐行读取 pi stdout，按 type 字段分派：

pi stdout type处理方式说明
"response"oneshot channel 唤醒匹配 id → 通过 pending_commands[id] 的 tx 发送 → send_command 的 rx 收到 → 返回给前端
"extension_ui_request"emit("pi-extension-ui-request")扩展请求用户交互（select/confirm/input/editor/notify/setStatus）
其他 (event)emit("pi-event")agent_start/end, message_update, tool_execution, compaction, queue_update 等
5

Event Tauri Event → 前端 EventConsumer

前端通过 listen("pi-event") 和 listen("pi-extension-ui-request") 捕获 Rust emit 的所有事件

handlePiEvent() 按事件类型分发到不同的 Zustand store：

事件类型目标 Store动作
agent_startSessionStore + MessageStoresetStreaming(true) + ensureAssistantMessage()
message_update (text_delta)MessageStorequeueMessageDelta → rAF 批量合并 → applyBatchDeltas
message_update (toolcall_start)MessageStoreaddToolCall(contentIndex, partial)
tool_execution_start/endMessageStoreupdateToolExecution(toolCallId, status, result)
agent_endSessionStore + MessageStoresetStreaming(false) + completeMessage
extension_ui_request (select)UIStoresetExtensionDialog → 弹出选择对话框
extension_ui_request (notify)UIStoreaddToast (排除 TPS 指标)
extension_ui_request (setStatus)UIStoresetExtensionStatus(key, text)
6

TypeScript Zustand Store → React 重渲染

Store 状态更新触发 React 组件重渲染，用户在 ChatPanel 中看到：

  • 流式文本逐步追加（通过 requestAnimationFrame 批量合并避免高频渲染）
  • 工具调用卡片从 pending → running → success/error
  • 扩展状态栏实时更新
  • 通知 toast 自动弹出和消失

4. 关键设计

◆ 进程启动：pi --mode rpc

pi 进程以 RPC 模式启动，stdin/stdout 全部管道化，无终端交互。Windows 上用 CREATE_NO_WINDOW 隐藏控制台窗口。

// pi_process.rs
cmd_builder.arg("--mode").arg("rpc")
  .stdin(Stdio::piped())
  .stdout(Stdio::piped())
  .stderr(Stdio::piped())
  .kill_on_drop(true);

◆ 请求-响应匹配：oneshot channel

每个命令注册一个 oneshot channel，通过 id 匹配 response。stdout_reader_task 收到 response 后唤醒对应的 send_command。

// commands.rs
let (tx, rx) = oneshot::channel();
pending_commands.insert(id, tx);

// 写入 stdin
stdin.write_all(json).await;
stdin.flush().await;

// 等待响应（带超时）
let response = timeout(duration, rx).await?;

◆ 流式渲染：rAF 批量合并

高频的 text_delta 事件不直接触发 store 更新，而是通过 requestAnimationFrame 批量合并后一次性 applyBatchDeltas，减少 React 渲染次数。

// tauri.ts
function queueMessageDelta(msgId, ci, field, delta) {
  blockQueue[field] += delta;
  if (!queuedMessageFrame) {
    queuedMessageFrame = requestAnimationFrame(
      flushQueuedMessageUpdates
    );
  }
}

// 单次 set() 完成所有 delta 追加
useMessageStore.getState().applyBatchDeltas(updates);

◆ 气泡生命周期

assistant 消息气泡由 agent_start/agent_end 控制生命周期，而非 message_update 的 done 事件。因为 toolUse 停止后 agent 会继续下一轮。

agent_start → ensureAssistantMessage()
  → message_update (text/toolcall 循环)
  → message_update done(reason="toolUse")
  → 继续下一轮... (不关闭气泡)
agent_end → completeMessage()

◆ 进程健康与自动重启

stdout EOF 时 Rust 清理全部状态并通知前端。前端监听 pi-process-exit 事件，1 秒后尝试自动重启。

// stdout_reader_task (EOF)
stdin = None;  child = None;
pending.drain() → "pi process exited"
emit("pi-process-exit", {reason:"eof"})

// 前端
listen("pi-process-exit", async () => {
  await sleep(1000);
  await startPi();  // 自动重启
});

◆ 超时策略

不同类型的命令有不同的超时时长，避免轻量命令被长时间阻塞，也允许 prompt 等命令有足够等待时间。

fn command_timeout(cmd_type: &str) -> Duration {
  match cmd_type {
    "prompt" | "new_session"
    | "switch_session" | "set_model"
      => 180s,
    "get_state" | "get_messages"
    | "get_available_models"
      => 10s,
    _ => 30s,
  }
}

◆ pi 可用性检测

启动时按优先级检测 pi：用户指定路径 → 配置文件路径 → 自动查找 (scoop/npm/where) → PATH。同时检测 bash 可用性（Windows 特有）。

1. 用户本次传入的路径
2. config.json 中的 pi_path
3. 自动查找:
   - $SCOOP/apps/pi-coding-agent/current/pi.exe
   - ~/AppData/Roaming/npm/pi.cmd
   - where pi
4. PATH 中的 pi/pi.cmd/pi.exe

◆ 会话扫描

Rust 扫描 ~/.pi/agent/sessions/ 目录，解析 JSONL 文件提取 session_info（name/pinned），5 秒 TTL 缓存。前端侧边栏展示 Project → Session 两级结构。

~/.pi/agent/sessions/
  --E--workspace-github-pi_desktop--/
    session-abc123.jsonl
    session-def456.jsonl

decode: --E--workspace-...-- → E:\workspace\...

◆ 配置持久化

应用配置保存到 ~/.config/pi-desktop/config.json。前端通过防抖 (300ms) 调用 set_app_config 持久化 UI 偏好。

AppConfig {
  window: { x, y, width, height, maximized },
  sidebar_width, sidebar_collapsed,
  properties_panel_open,
  theme, font_family, font_size,
  last_session, pi_path,
}
