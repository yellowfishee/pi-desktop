# 步骤 5：完整 Git 工作流

## 目标

让 Changes 面板支持查看、暂存、取消暂存、放弃和提交，并在每次操作后显示最新状态。

## 接口设计

- 所有 Git 命令接收经过后端验证的真实 `projectPath`。
- 文件参数始终通过 `--` 传递，不拼接 shell 命令。
- 放弃变更必须区分已跟踪文件和未跟踪文件，并在前端二次确认。

## 实施链路

1. 支持没有 `HEAD` 的新仓库：分别读取工作区、索引和未跟踪文件。
2. 使用适合机器解析的 Git 输出，正确处理空格、引号、重命名和非 UTF-8 路径边界。
3. 在文件行提供 stage、unstage 和 discard 操作。
4. 接入提交对话框，支持提交已暂存文件和明确选择的文件。
5. 添加手动刷新按钮。
6. 操作成功后刷新变更、分支和 diff；失败时保留当前选择并显示错误。
7. 工具调用结束、窗口重新聚焦和项目切换后触发受控刷新。

## 验收标准

- 新仓库首次提交前可以查看和暂存文件。
- 已暂存、未暂存、同时存在两类变更的文件状态正确。
- 重命名、删除、未跟踪和二进制文件不会破坏列表解析。
- 放弃操作必须确认，且不会误删未选择文件。
- 提交完成后显示新提交哈希并刷新面板。
- 所有 Git 操作都在所选真实项目路径执行。
